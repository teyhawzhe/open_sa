# 風險評估規則

## 觸發時機
- 每次新增需求時，AI 必須主動產生初步風險評估。
- 每次修改既有需求時，AI 必須重新評估本次異動造成的風險。
- 風險評估不只是提醒，AI 必須先提出自己的分析，再標示需要人工確認的項目。

## 風險項目
AI 至少評估：
- 是否影響既有業務流程。
- 是否影響使用者角色、權限或資料存取。
- 是否影響既有 SRS、Wireframe、UML、API、DB Table 或資料模型。
- Wireframe 是否新增 SRS 未提到的欄位、按鈕、流程、狀態或錯誤訊息。
- Wireframe 是否影響 API request / response、DB Table 欄位、權限規則或驗收條件。
- 是否造成資料庫 migration、資料轉換、索引、約束、效能或相容性風險。
- 是否產生新的例外情境。
- 是否影響驗收條件或測試案例。
- 是否存在未確認假設。
- 是否可能造成需求範圍擴大。
- 是否有法規、安全性、效能或維運風險。

## 風險等級
- `低`：影響範圍小，通常不影響既有流程或介面。
- `中`：可能影響流程、角色、資料、測試或文件一致性，需要人工確認。
- `高`：可能影響核心流程、權限、安全、法規、資料正確性或既有系統相容性，必須人工確認後才能繼續。

## 輸出格式
```md
# 風險評估：REQ-001

## 本次異動摘要
說明這次需求新增或修改了什麼。

## AI 初步風險分析
| 風險項目 | 風險等級 | 說明 | 建議處理方式 | 是否需人工確認 |
|---|---|---|---|---|
| 權限影響 | 中 | 新增登入流程可能影響角色驗證 | 確認角色與權限規則 | 是 |

## AI 提醒
列出本次最需要使用者注意的 3 到 5 個風險或待確認事項。
```

## 寫入規則
- 風險評估結果寫入該需求的 `risk.md`。
- `history.md` 需記錄本次異動的整體風險等級。
- 若有高風險項目，相關需求文件狀態應維持 `待確認`。
