# DB Table 設計規則

## 基本原則
- DB Table 設計必須依據已確認或明確標示待確認的需求產出。
- 不可自行新增未在需求中出現的資料欄位、狀態或業務規則。
- 每個資料表、欄位、索引與約束都要能追溯到需求或明確的技術理由。
- 若資料庫類型未指定，先使用通用關聯式資料庫描述，不綁定特定 SQL 方言。

## 輸出文件
DB Table 設計使用 `db-table.md`，放在對應需求資料夾內。

```text
project/
  REQ-001-login/
    db-table.md
```

## 文件欄位
`db-table.md` 至少包含：
- 需求編號。
- 狀態：`草稿`、`待確認`、`已審核`。
- 設計目的。
- 適用範圍。
- Table 清單。
- 欄位定義。
- Primary Key。
- Foreign Key。
- Index。
- Unique Constraint。
- 預設值。
- Nullable 規則。
- 資料保留或刪除規則。
- Migration 影響。
- 待確認事項。

## Table 定義格式
```md
## Table：users

### 用途
說明此資料表支援哪個需求或業務流程。

### 欄位
| 欄位名稱 | 型別 | 必填 | 預設值 | 說明 | 來源需求 |
|---|---|---|---|---|---|
| id | bigint | 是 | 無 | 使用者識別碼 | REQ-001 |

### Key 與 Constraint
| 類型 | 名稱 | 欄位 | 說明 |
|---|---|---|---|
| PK | pk_users | id | 使用者主鍵 |

### Index
| 名稱 | 欄位 | 唯一 | 用途 |
|---|---|---|---|
| idx_users_email | email | 是 | 登入查詢 |
```

## 欄位設計規則
- 欄位名稱需語意清楚，避免縮寫造成誤解。
- 每個欄位都要有說明與來源需求。
- 必填欄位需說明為何不可為空。
- 狀態欄位需列出允許值與狀態轉換來源。
- 金額、時間、時區、語系、個資欄位需特別標示。
- 敏感資料不得只寫「加密」，需標示保護需求為待確認或已確認。

## Migration 與相容性
若修改既有 Table，必須說明：
- 新增、修改或刪除哪些 Table / 欄位 / Index / Constraint。
- 是否需要資料轉換。
- 是否影響既有 API、報表、批次或權限。
- 是否有 rollback 需求。
- 是否可能造成停機、鎖表或效能風險。

## 待確認事項
以下情況必須標示待確認：
- 欄位型別無法從需求判斷。
- 是否允許 null 不明確。
- 唯一性規則不明確。
- 刪除規則不明確。
- 個資、敏感資料或稽核需求不明確。
- 資料保存期限不明確。
